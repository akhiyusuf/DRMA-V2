# DRMA Supabase Storage — Bucket Policy & Naming Convention

> LOW-03 fix: Documents the storage bucket architecture established after
> the security audit. All new code that uploads files to Supabase storage
> MUST follow this convention.

## Buckets

| Bucket | Visibility | Purpose | Size Limit | Allowed Types |
|---|---|---|---|---|
| `media` | **Public** | Marketing assets — product images, hero images, CMS-managed content. World-readable via direct URL. | None | None (recommend images only) |
| `user-uploads` | **Private** | Customer-supplied files — order attachments, support tickets, profile photos. NOT world-readable; access only via signed URLs generated by an authenticated API route. | 10 MB | `image/jpeg`, `image/png`, `image/webp`, `image/gif` |

## Naming Convention

### Public `media` bucket

Files are named by the CMS dashboard upload handler. The current pattern is:

```
{unix_timestamp_ms}-{original_filename}.{ext}
```

Examples:
- `1781797656125-hero_main.png`
- `1782444948805-07820C24-CD42-4681-81A0-11E9F1C13987.jpeg`

**Going forward**, prefer descriptive, kebab-case names without timestamps when
the file is managed (not user-uploaded):

```
product-{product-id}-{view}.{ext}     ← e.g. product-p1-front.png
hero-{section}.{ext}                   ← e.g. hero-main.png, hero-mission.png
cms-{section}-{purpose}.{ext}          ← e.g. cms-about-founder.png
```

### Private `user-uploads` bucket

Files MUST be named with a random prefix to prevent enumeration and collisions:

```
{random_16_char_nanoid}-{user_id}-{purpose}.{ext}
```

Examples:
- `V1StGXR8_Z5jHi2p-user_abc123-order_attachment.png`
- `aB3dEf7Gh9JkLmNoP-user_xyz789-support_ticket.jpg`

The random prefix ensures that even if a signed URL leaks, the filename
cannot be guessed to access other uploads.

## Access Patterns

### Reading from `media` (public)

Direct URL, no auth needed:
```
https://qeyfzpbbukhnuiabrkef.supabase.co/storage/v1/object/public/media/{filename}
```

Used in `<img src="...">` tags. Covered by CSP `img-src 'https:'`.

### Reading from `user-uploads` (private)

MUST go through an authenticated Next.js API route that generates a
short-lived signed URL:

```typescript
// Example: src/app/api/user/uploads/[id]/route.ts
import { supabaseAdmin } from "@/utils/supabase/admin";
import { requireCmsAuth } from "@/utils/cms-auth";

export async function GET(request, { params }) {
  // Verify the requester is authorized to see this file
  const authError = requireCmsAuth(request);
  if (authError) return authError;

  const { data, error } = await supabaseAdmin
    .storage
    .from("user-uploads")
    .createSignedUrl(params.id, 300); // 5-minute expiry

  if (error) return Response.json({ error: "Not found" }, { status: 404 });
  return Response.json({ url: data.signedUrl });
}
```

Never expose the service role key to the client. All signed-URL generation
happens server-side.

### Writing to `media` (CMS only)

Use the existing `POST /api/cms/upload` endpoint (requires CMS auth).
Files are stored with a timestamp prefix.

### Writing to `user-uploads`

Create a new authenticated endpoint (e.g., `POST /api/user/uploads`) that:
1. Validates the user's session (Supabase Auth or CMS auth, depending on use case)
2. Validates file type and size (defense-in-depth, even though the bucket enforces limits)
3. Generates a random filename per the convention above
4. Uploads via `supabaseAdmin.storage.from("user-uploads").upload(...)`
5. Records the upload in a `user_uploads` table (file path, uploader ID, created_at)
